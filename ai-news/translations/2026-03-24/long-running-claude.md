# 科学计算中的长时间运行 Claude

> 原文：[Long-running Claude for scientific computing](https://www.anthropic.com/research/long-running-Claude)
> 作者：Siddharth Mishra-Sharma
> 译者：Alice Larry

---

在这篇文章中，Discovery 团队的研究员 Siddharth Mishra-Sharma 解释了如何将多天的智能体编码工作流——测试预言机、持久化记忆和编排模式——应用到即使在自己专业领域之外的科学计算任务中。

## 前提

目前使用 AI 智能体的大多数科学家以对话循环的方式工作，紧密管理过程中的每一步。随着模型在过去一年左右[在长周期任务上显著改进](https://metr.org/time-horizons/)，一种新的工作方式出现了：与其参与每一个细节，我们可以指定高层目标，然后让一队智能体自主工作。这使得可以在几小时内完成可能需要我们几天、几周甚至几个月的项目。某些类型的科学任务非常适合这种模式，例如重新实现一个数值求解器、将用旧 Fortran 方言编写的遗留科学软件转换为现代语言，以及针对参考实现调试大型代码库。这些是工作范围明确、成功标准清晰、人工监督可以是偶发而非持续的任务。

Anthropic 的 [C 编译器项目](https://www.anthropic.com/engineering/building-c-compiler)展示了这种模式的一个版本，其中 Claude 跨越大约 2,000 个会话构建了一个能够编译 Linux 内核的 C 编译器。这篇文章描述了如何使用 Claude Code 为科学计算任务设置类似的模式，针对的是典型的学术实验室。作为一个具体例子，我将演示使用 Claude Opus 4.6 来[实现一个可微分版本的宇宙学 Boltzmann 求解器](https://github.com/smsharma/clax)。这是预测大爆炸余辉（宇宙微波背景辐射，或 CMB）统计特性的数值代码。它通过演化光子、重子、中微子和暗物质在早期宇宙中的耦合方程来实现这一点。

像 [CLASS](http://class-code.net/) 和 [CAMB](https://camb.info/) 这样的 Boltzmann 求解器是宇宙学中的核心科学基础设施，使我们能够使用来自 Planck 和 Simons Observatory 等巡天观测的数据来约束宇宙学模型。可微分版本——可以通过整个求解器传播梯度的版本——使得基于梯度的推断方法成为可能，大大加速参数估计。用 JAX 编写是一个自然的选择，因为它基本上免费给我们自动微分和与加速器（如 GPU）的兼容性。

值得注意的是，这个任务不在我的核心科学领域——我对工具和科学有高层次的了解，但没有在合理时间内自己完成它的专业知识。拥有这种专业知识的团队已经用 JAX 构建了具有 CLASS 中部分功能的[可微分](https://arxiv.org/abs/2311.03291)[求解器](https://arxiv.org/abs/2602.15104)。这些工作通常代表数月到数年的研究人员时间。这里的重点是看看智能体是否可以在非领域专家的最小指导下走得更远。

这类任务在结构上与 C 编译器项目不同，后者可以分配给大量并行智能体。而 Boltzmann 求解器是一个深度耦合的管道——在建模早期宇宙如何复合时的小数值误差或不良近似可以微妙地影响下游的一切。因此它需要不同的智能体技能。调试需要因果追踪整个链条并利用领域知识，这可能更适合单个智能体顺序工作，根据需要生成子智能体，并使用参考实现来二分差异。

我们将使用运行 SLURM 作业调度器的 HPC 集群作为计算环境，但核心思想——进度文件、测试预言机、带有明确规则的智能体提示——无论你在哪里运行 Claude Code 都适用。

## 起草计划并在本地迭代

在向管理自主研究智能体团队的转变中，你应该花大部分时间（与 Claude 协商）制定一套清晰阐述项目交付物和相关背景的指令。这些指令应该放在位于根目录的 CLAUDE.md 文件中。Claude 特别对待这个文件，将其保持在上下文中并参考它来了解整体计划。关键的是，Claude 可以在工作时编辑这些指令，在处理问题时为未来的工作更新它们。

[这里](https://github.com/smsharma/clax/blob/6a6b2330cf25edded1bb31ec57a0091aa794a5d3/CLAUDE.md)是宇宙学 Boltzmann 求解器项目的早期 CLAUDE.md，显示了在初次尝试编写求解器后编纂的整体计划和设计决策。为了达到这一点，我指定了项目的高层目标——实现与参考 CLASS 实现的完全功能一致同时保持完全可微分，并且在主要科学交付物上达到与 CLASS 0.1% 的精度目标——并与 Claude 迭代直到计划看起来令人满意。鉴于 0.1% 是两个规范 Boltzmann 代码 CLASS 和 CAMB 之间的典型一致水平，这似乎是一个好的科学目标。

## 跨会话的记忆

进度文件（按惯例我们称之为 CHANGELOG.md）是智能体的便携式长期记忆，作为一种实验室笔记。在 CLAUDE.md 中，Claude 被指示在这个文件中跟踪进度。

一个好的进度文件可能会跟踪当前状态、已完成的任务、失败的方法及其原因、关键检查点的精度表格，以及已知的限制。失败的方法很重要——没有它们，连续的会话会重新尝试相同的死胡同。一个条目可能看起来像："尝试对扰动 ODE 使用 Tsit5，系统太刚性。切换到 Kvaerno5。"[这里](https://github.com/smsharma/clax/blob/main/CHANGELOG.md)是运行示例的变更日志，显示了这些元素。

## 测试预言机

虽然更开放式的科学发现通过智能体确实在地平线上，但今天长时间运行的自主科学工作至关重要地依赖于智能体有办法知道它是否在取得进展。对于科学代码，这可能是参考实现、明确可量化的目标或现有的测试套件。指示智能体扩展测试套件并在工作时运行测试以防止回归也很有帮助。在我的示例任务中，Claude 被指示使用 [CLASS C 源代码](https://github.com/lesgourg/class_public)作为参考实现来构建和持续运行单元测试。

## Git 作为协调工具

Git 可以是以免干预方式监控和协调智能体进度的好方法。智能体应该在每个有意义的工作单元之后提交和推送。这给你一个可恢复的历史记录，以防出现问题，使进度在本地可见，并防止工作丢失，例如，如果你的计算配额在会话中途用完。

实际上，这可能是 CLAUDE.md 中的一组指令，例如"在每个有意义的工作单元之后提交和推送。在每次提交之前运行 `pytest tests/ -x -q`。永远不要提交破坏现有通过测试的代码。"

要引导智能体，你总是可以 SSH 进入集群并手动重新提示和/或更新其指令。通常更方便的是简单地让本地的 Claude Code 实例 SSH 进去并为你运行命令；这也适用于下面描述的所有内容。

## 执行循环

如上所述，在你有一个看起来合理并编码在 CLAUDE.md 中的计划之前，首先在本地迭代计划通常很有用。从那里，在计算节点上的终端复用器（如 tmux）中启动一个 Claude Code 会话，告诉智能体在哪里找到你的代码库，然后让它工作。因为会话在 tmux 中运行，你可以断开连接，合上笔记本电脑，偶尔检查进度（对于 Boltzmann 求解器，我会在 GitHub 上用手机检查，例如，在排队买咖啡时）。

在 HPC 集群上，你可能通过 SLURM 调度器请求一个节点，一个启动 Claude Code 在 tmux 会话中的示例作业脚本可能如下所示：

```bash
#!/bin/bash
#SBATCH --job-name=claude-agent
#SBATCH --partition=GPU-shared
#SBATCH --gres=gpu:h100-32:1
#SBATCH --time=48:00:00
#SBATCH --output=agent_%j.log
cd $PROJECT/my-solver
source .venv/bin/activate
export TERM=xterm-256color
tmux new-session -d -s claude "claude; exec bash"
tmux wait-for claude
```

一旦作业开始，你附加到 tmux 会话，给 Claude Code 指示（例如，"阅读 CHANGELOG.md 并开始下一个任务"），当你满意它在正确的轨道上时断开连接。你可以随时重新附加以检查、引导或开始新任务，使用类似这样的命令：

```bash
srun --jobid=JOBID --overlap --pty tmux attach -t claude
```

### Ralph 循环

随着模型变得更强大，它们需要更少的定制编排，如提示工程、RAG 或上下文填充。然而，在任何给定时间点，提供一定程度的脚手架作为能力提升可能是有用的。例如，当前模型可能会出现智能体懒惰——当被要求完成复杂的多部分任务时，它们有时会找借口在完成整个任务之前停止（"太晚了，我们明天再继续？"）。

为了规避这个问题，一个有用的编排模式是 [Ralph 循环](https://ghuntley.com/loop/)，它本质上是一个 for 循环，当智能体声称完成时将其踢回上下文，并询问它是否真的完成了。这对于长时间运行的任务很有用，因为智能体会承认任务没有达到标准，并继续工作直到达到标准。其他类似的模式包括 [GSD](https://github.com/gsd-build/get-shit-done)（以及[特定领域的](https://arxiv.org/abs/2603.20179)[变体](https://github.com/psi-oss/get-physics-done)）以及 Claude Code 原生的 /loop 命令。

Ralph 可以通过 /plugin 安装。在 Claude Code 中的典型调用提示可能如下：

```
/ralph-loop:ralph-loop "请继续工作直到达到整个参数范围内 0.1% 精度的成功标准。" --max-iterations 20 --completion-promise "DONE"
```

在这里，Claude 将迭代最多 20 次，直到它用"DONE"咒语保证任务完成。

## 结果

Claude 从头开始在[这个项目](https://github.com/smsharma/clax)上工作了几天，在参考 CLASS 实现的各种输出上达到了次百分比的一致性。我让 Claude 重建项目过程中一些主要输出——各种 CMB 角功率谱——的精度，同时标记开发过程中的里程碑。它生成了下面的图，说明了达到次百分比精度的路径。

> 随着智能体在代码库上工作，随时间达到次百分比精度的路径。

智能体的开发轨迹有些笨拙。例如，它的测试覆盖率有明显的差距——有一段时间它只在单个（基准）参数点测试代码，大大减少了其捕获错误的覆盖面。它也会犯基本错误，比如在规范约定上绊倒，或花几个小时追踪宇宙学家会立即发现的错误，但它持续朝着规定的次百分比精度目标取得进展。

这个项目的一个副作用是，我通过观看 git 提交历史学到了令人惊讶的关于 Boltzmann 求解器及其建模的物理的知识。这个项目不在我的核心科学领域，但跟随 Claude 的增量进展并查找我不认识的内容被证明是一种有效的渗透科学的方式。[提交日志](https://github.com/smsharma/clax/commits/main/)读起来像一个快速、超级字面意思的博士后的实验室笔记。

虽然生成的求解器还没有达到生产级（例如，它在每个范围内都没有达到与参考 CLASS 实现可接受的精度匹配），但它表明智能体驱动的开发可以将数月甚至数年的研究人员工作压缩到几天内。

这种压缩改变了什么算作空闲时间。AI 研究中的一个普遍经历是晚上启动一个实验（例如，一次训练运行），然后第二天早上满意地看到结果。不运行实验有机会成本。如今，不运行智能体感觉也有成本。如果你有计算资源和具有明确成功标准的项目，每一个你没有智能体为你工作的夜晚都是潜在的进展被搁置。
