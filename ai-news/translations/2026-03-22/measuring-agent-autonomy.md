## 在真实环境中研究智能体

智能体难以进行实证研究。首先，关于智能体*是什么*，目前尚无公认定义。其次，智能体发展迅速。去年，许多最复杂的智能体——包括 Claude Code——还只涉及单一对话线程，而如今已有多智能体系统能自主运行数小时。最后，模型提供商对其客户的智能体架构能见度有限。例如，我们无法可靠地将对 API 的独立请求归并为连贯的智能体活动"会话"。（我们在本文末尾对这一挑战有更详细的讨论。）

面对这些挑战，我们如何对智能体进行实证研究？

首先，我们为本研究采用了一个概念上有据可查且可操作的智能体定义：*智能体是配备了工具从而能够采取行动的 AI 系统*，这些行动包括运行代码、调用外部 API、向其他智能体发送消息等。研究智能体使用的工具，能让我们深入了解它们在现实世界中的行为。

其次，我们建立了一套指标体系，数据来源涵盖公开 API 的智能体使用情况和我们自研的编程智能体 Claude Code。两者在广度与深度上各有取舍：

- **公开 API** 让我们能够广泛了解数千个不同客户的智能体部署情况。我们没有试图推断客户的智能体架构，而是在*单次工具调用*层面进行分析。这一简化假设使我们能够对真实世界的智能体作出有据可查、一致的观察，即便各智能体的部署背景差异显著。这种方法的局限在于，我们必须孤立地分析每个动作，无法还原单个动作如何随时间组合成更长的行为序列。

- **Claude Code** 提供了相反的权衡。由于 Claude Code 是我们自己的产品，我们可以关联跨会话的请求，从头到尾理解完整的智能体工作流程。这使 Claude Code 在研究自主性方面尤为有用——例如，智能体在无人干预的情况下能运行多久、什么触发了中断，以及用户如何随着经验积累维持对 Claude 的监督。然而，由于 Claude Code 只是一款产品，它无法像 API 流量那样提供对智能体使用情况的多样化洞察。

通过在隐私保护基础设施的支撑下综合运用两个数据源，我们能够回答任何单一来源都无法独立解答的问题。

## Claude Code 自主运行的时间越来越长

智能体在无人参与的情况下实际能运行多久？在 Claude Code 中，我们可以逐轮追踪 Claude 从开始工作到停止（无论是因为完成任务、提问，还是被用户打断）所经历的时间，从而直接测量这一指标。

轮次时长是自主性的不完美代理指标。例如，能力更强的模型可以更快完成同样的工作，子智能体则允许更多工作同时进行，这两者都会使轮次时长缩短。与此同时，用户可能随着时间推移尝试更有野心的任务，这会推动轮次时长增加。此外，Claude Code 的用户群正在快速增长，因而也在不断变化。我们无法单独衡量这些变化；我们测量的是这些因素相互作用的净结果，包括用户允许 Claude 独立工作的时长、任务难度，以及产品本身的效率（后者每日都在提升）。

大多数 Claude Code 轮次都很短暂。中位轮次时长约为 45 秒，在过去几个月中仅有轻微波动（在 40 至 55 秒之间）。事实上，第 99 百分位以下的几乎每个百分位都保持相对稳定。这种稳定性正是我们对一款快速增长产品的预期：新用户采用 Claude Code 时相对缺乏经验，而且——正如我们在下一节所示——他们不太可能给予 Claude 充分的自由度。

更具启示性的信号在于尾部。最长的轮次最能揭示 Claude Code 最雄心勃勃的使用方式，也指向自主性的发展方向。2025 年 10 月至 2026 年 1 月间，第 99.9 百分位的轮次时长几乎翻了一番，从不足 25 分钟增至超过 45 分钟（图 1）。

**图 1.** 交互式 Claude Code 会话中第 99.9 百分位轮次时长（Claude 单轮工作时长），7 日滚动平均。第 99.9 百分位从 9 月底的不足 25 分钟稳步增长至 1 月初的超过 45 分钟。本分析反映所有交互式 Claude Code 的使用情况。

值得注意的是，这一增长在各次模型发布间呈平滑态势。如果自主性纯粹是模型能力的函数，我们会预期每次新模型发布时出现陡峭跳升。这一趋势的相对平稳性，反而暗示有几个潜在因素共同作用，包括：高级用户随时间推移对工具建立起信任、将 Claude 应用于日益有野心的任务，以及产品本身的持续改进。

自 1 月中旬以来，极端轮次时长有所下降。我们对此推测有几点原因。第一，Claude Code 的用户群在 1 月至 2 月中旬之间翻了一番，更庞大、更多元的会话群体可能重塑了分布形态。第二，随着用户从假期返回，他们带来 Claude Code 的项目可能从兴趣项目转向了边界更清晰的工作任务。最可能的是，这些因素以及其他我们尚未识别的因素共同作用的结果。

我们还研究了 Anthropic 内部的 Claude Code 使用情况，以理解独立性与实用性如何协同演进。从 8 月到 12 月，Claude Code 在内部用户最具挑战性任务上的成功率翻了一番，与此同时，每次会话的平均人工干预次数从 5.4 次降至 3.3 次。用户正在给予 Claude 更多自主权，而至少在内部，他们在减少干预的同时取得了更好的结果。

这两项测量都指向一个显著的**部署滞后现象**：模型所能驾驭的自主程度超出了其在实践中实际发挥的水平。

将这些发现与外部能力评估进行对比颇具意义。最广为引用的能力评估之一是 METR 的《测量 AI 完成长任务的能力》，该报告估计 Claude Opus 4.5 能以 50% 的成功率完成需要人类花费近 5 *小时*才能完成的任务。相比之下，Claude Code 中第 99.9 百分位的轮次时长约为 42 分钟，中位数则更短。然而，这两个指标并不能直接比较。METR 评估捕捉的是模型在理想化环境中——无人工交互、无真实后果——的能力上限。我们的测量捕捉的是实践中发生的情况，其中 Claude 会暂停寻求反馈，用户也会中断。而且 METR 的五小时数字衡量的是任务难度——人类完成该任务需要多长时间——而非模型实际运行时长。

能力评估与我们的测量单独来看都无法给出智能体自主性的完整图景，但合而观之，它们表明在实践中赋予模型的自由度落后于模型实际能够驾驭的程度。

## 经验丰富的 Claude Code 用户更频繁地使用自动审批，但也更频繁地打断操作

人类如何随时间适应与智能体的协作方式？我们发现，随着使用经验的积累，人们给予 Claude Code 的自主权越来越大（图 2）。较新的用户（会话数 <50）约有 20% 的时间采用完全自动审批；到 750 次会话时，这一比例上升至超过 40%。

这一转变是渐进的，表明信任在稳步积累。同样值得注意的是，Claude Code 的默认设置要求用户手动审批每项操作，因此部分转变可能反映的是用户在熟悉 Claude 能力后，根据自己对更大独立性的偏好来配置产品。

**图 2.** 按账号使用时长划分的自动审批率。经验丰富的用户越来越多地让 Claude 在无需任何手动审批的情况下运行。数据反映 2025 年 9 月 19 日之后注册用户的所有交互式 Claude Code 使用情况。折线和置信区间经 LOWESS 平滑处理（带宽 0.15）。横轴为对数刻度。

审批操作只是监督 Claude Code 的方式之一。用户还可以在 Claude 工作时中断以提供反馈。我们发现，中断率随经验增长而提高。新用户（约有 10 次会话）在 5% 的轮次中中断 Claude，而经验丰富的用户的中断率约为 9%（图 3）。

**图 3.** 按账号使用时长划分的逐轮中断率。经验丰富的用户中断 Claude 的频率更高，而非更低。数据反映 2025 年 9 月 19 日之后注册用户的所有交互式 Claude Code 使用情况。阴影区域显示 95% Wilson 分数置信区间。折线和置信区间经 LOWESS 平滑处理（带宽 0.15）。横轴为对数刻度。

中断次数*和*自动审批率都随经验增长而提高。这一表面矛盾反映了用户监督策略的转变。新用户更倾向于在每项操作执行前逐一审批，因此很少需要在 Claude 执行过程中中途打断。经验丰富的用户则更倾向于让 Claude 自主运行，仅在出现问题或需要调整方向时介入。更高的中断率也可能反映出有经验的用户进行了积极监控——他们对何时需要介入有着更敏锐的直觉。我们预计逐轮中断率最终会趋于平稳，随着用户形成稳定的监督风格，这一曲线实际上可能已经在最有经验的用户群体中开始趋于平缓（尽管会话次数较多时置信区间变宽，使这一点难以确认）。我们在公开 API 上也看到了类似模式：在最低复杂度任务（如修改一行代码）中，87% 的工具调用涉及某种形式的人工参与；而在高复杂度任务（如自主发现零日漏洞或编写编译器）中，这一比例仅为 67%。

综合来看，这些发现表明经验丰富的用户并不一定是在放弃监督。中断率随经验增长而提高（与自动审批率同步增长），表明存在某种形式的主动监控。这印证了我们此前强调的一点：有效的监督不在于审批每一项操作，而在于在关键时刻具备介入的能力。

## Claude Code 主动暂停寻求澄清的频率高于人类打断它的频率

当然，影响自主性在实践中如何展开的并非只有人类。Claude 本身也是积极的参与者，当它不确定如何推进时，会主动停下来寻求澄清。我们发现，随着任务复杂性的增加，Claude Code 寻求澄清的频率也在上升——而且比人类主动中断它的频率更高（图 4）。

**图 4.** Claude 提出的澄清问题与人类的中断，按目标复杂度划分。随着任务变得更复杂，Claude 更有可能寻求澄清，人类也更有可能中断。Claude 主动中止的增速快于人类主动中止。所有类别的 95% CI < 0.9%，n = 50 万次交互式 Claude Code 会话。

在最复杂的任务中，Claude Code 请求澄清的频率是最低复杂度任务的两倍以上，这表明 Claude 对自身不确定性有一定的校准能力。然而，重要的是不应过度解读这一发现：Claude 未必总在正确的时机停下来，它可能会提出不必要的问题，其行为也可能受到"计划模式"（Plan Mode）等产品功能的影响。不管怎样，随着任务难度增加，Claude 越来越多地通过主动停下来征询人类意见来限制自己的自主性，而不是等待人类介入。

## 智能体已进入高风险领域，但尚未规模化

人们在将智能体用于什么？这些部署有多大风险？这些智能体有多高的自主性？风险与自主性之间存在权衡吗？

为回答这些问题，我们使用 Claude 对公开 API 中各次工具调用的相对风险和自主程度进行 1 到 10 分的评估。简而言之，风险评分为 1 反映出错后无任何后果的操作，风险评分为 10 则涵盖可能造成重大危害的操作。我们用同样的量表评估自主性，低自主性意味着智能体似乎在遵循明确的人工指令，高自主性意味着它在独立运作。然后，我们将相似的操作归入同一聚类，并计算每个聚类的平均风险和自主性评分。

**表 2.** 平均风险或平均自主性较高的聚类。基于公开 API 的 998,481 次随机工具调用样本。

| 平均风险较高 | 平均自主性较高 |
|---|---|
| 以合法开发功能为掩护实施 API 密钥窃取后门（风险：6.0，自主性：8.0） | 以合法开发为掩护进行红队特权提升和凭证窃取（自主性：8.3，风险：3.3） |

| 为请求用户检索并显示患者病历（风险：4.4，自主性：3.2） | 自主执行加密货币交易以获取利润（自主性：7.7，风险：2.2） |

鉴于我们对这些智能体的能见度有限，这些操作有可能是模拟的（例如，我们无法确定 Claude 下达的金融交易指令是否真的被执行，而且我们相信许多"数据窃取"行为实际上是评估测试）。尽管这些高风险操作在总流量中占比很少，但单次错误的后果仍可能十分严重。

我们发现，80% 的工具调用来自似乎至少具备一种保障机制的智能体（如受限权限或人工审批要求），73% 似乎在某种程度上有人工参与，仅有 0.8% 的操作似乎是不可逆的（如向客户发送邮件）。

## 局限性

这项研究只是一个开端。我们仅提供了对智能体活动的局部视角，我们希望坦诚地说明我们的数据能够揭示什么、不能揭示什么：

- 我们只能分析来自单一模型提供商——Anthropic——的流量。基于其他模型构建的智能体可能呈现出不同的采用模式、风险特征和交互动态。

- 我们的两个数据来源提供了互补但不完整的视角。公开 API 流量让我们对数千个部署有广泛了解，但我们只能孤立地分析单次工具调用，而非完整的智能体会话。Claude Code 提供了完整的会话数据，但仅限于一款主要用于软件工程的产品。我们许多最有力的发现都基于 Claude Code 的数据，可能无法推广到其他领域或产品。

- 我们的分类由 Claude 生成。我们为每个维度提供了退出类别（如"无法推断"、"其他"），并尽可能与内部数据进行验证（详见附录），但由于隐私限制，我们无法手动检查底层数据。某些保障机制或监督机制也可能存在于我们能观察到的上下文之外。

- 本分析反映的是特定时间窗口（2025 年底至 2026 年初）的情况。智能体领域正在快速变化，随着能力增长和采用演进，相关模式可能会发生变化。我们计划随时间延伸这一分析。

- 我们的公开 API 样本在单次工具调用层面抽取，这意味着涉及大量连续工具调用的部署（如软件工程工作流中重复的文件编辑）相对于以更少操作实现目标的部署而言被过度代表。这种抽样方式反映的是智能体活动的*量*，而非智能体部署或用途的分布。

- 我们研究 Claude 在公开 API 上使用的工具及这些操作的上下文，但对客户在公开 API 之上构建的更广泛系统能见度有限。一个在 API 层面看起来自主运作的智能体，可能在下游有我们无法观察到的人工审查环节。尤其是，我们对风险、自主性和人工参与的分类反映的是 Claude 从单次工具调用上下文中推断出的内容，无法区分在生产环境中执行的操作和作为评估或红队测试执行的操作。风险最高的几个聚类似乎是安全评估，这凸显了我们对每个操作所处更广泛背景的能见度的局限性。

## 展望未来

我们正处于智能体采用的早期阶段，但自主性正在增强，高风险部署正在涌现，尤其是随着 Cowork 等产品使智能体更易获取。下文我们为模型开发者、产品开发者和政策制定者提供建议。鉴于我们才刚刚开始在真实环境中测量智能体行为，我们避免提出强硬处方，而是着重指出未来工作的方向。

**模型和产品开发者应投资于部署后监控。** 部署后监控对于理解智能体的实际使用方式至关重要。部署前评估测试的是智能体在受控环境中的能力，但我们的许多发现仅凭部署前测试是无法观察到的。除了理解模型的能力之外，我们还必须理解人们在实践中如何与智能体交互。我们在此报告的数据之所以存在，是因为我们*选择*构建了收集这些数据的基础设施。但还有更多工作要做。我们没有可靠的方法将公开 API 的独立请求关联成连贯的智能体会话，这限制了我们在 Claude Code 等第一方产品之外对智能体行为的了解。以保护隐私的方式开发这些方法，是跨行业研究与合作的重要方向。

**模型开发者应考虑训练模型识别自身的不确定性。** 训练模型识别自身的不确定性并主动向人类反映问题，是一种重要的安全属性，能够与人工审批流程和访问限制等外部保障措施相辅相成。我们训练 Claude 做到这一点（我们的分析也表明 Claude Code 提问的频率高于人类中断它的频率），并鼓励其他模型开发者也这样做。

**产品开发者应为用户监督而设计。** 对智能体的有效监督不仅仅是在审批链中加入一个人。我们发现，随着用户积累使用智能体的经验，他们倾向于从审批每个单独操作转变为监控智能体的行为并在必要时介入。以 Claude Code 为例，经验丰富的用户自动审批更多，但也中断更多。我们在公开 API 上也看到了相关模式：随着目标复杂性的增加，人工参与度似乎有所下降。产品开发者应投资于能让用户可靠地了解智能体正在做什么的工具，以及允许他们在出错时重定向智能体的简便干预机制。这是我们持续为 Claude Code 投资的方向（例如，通过实时引导和 OpenTelemetry），我们也鼓励其他产品开发者同样如此。

**现在对特定交互模式进行强制规定为时过早。** 有一个领域，我们确实有信心提供明确指引，那就是*不*应强制规定什么。我们的发现表明，经验丰富的用户会从审批每个单独的智能体操作转向监控并在必要时介入。规定特定交互模式的监督要求——例如要求人类审批每项操作——会制造摩擦而不一定带来安全收益。随着智能体和智能体测量科学的成熟，重点应放在人类是否处于能够有效监控和介入的位置，而非要求特定形式的参与。

这项研究的核心教训是：智能体在实践中体现的自主性，是由模型、用户和产品共同构建的。Claude 通过在不确定时暂停提问来限制自己的独立性。用户在与模型协作的过程中建立信任，并相应调整监督策略。我们在任何部署中观察到的现象，都是这三股力量共同作用的结果，这也是为什么它不能完全由部署前评估来描述。理解智能体的实际行为，需要在真实世界中对其进行测量，而实现这一目标的基础设施仍处于萌芽阶段。
