# OpenAI 如何以低延迟大规模交付语音 AI

原文链接：https://openai.com/index/delivering-low-latency-voice-ai-at-scale/
发布日期：2026-05-04

> 说明：以下为对 OpenAI 官方文章的完整中文翻译，保留核心技术结构、术语与论证逻辑。

只有当对话速度接近人类语速时，语音 AI 才会显得自然。当网络成为瓶颈时，人们会立刻听见问题，表现为尴尬的停顿、被截断的打断，或延迟的插话响应。这对于 ChatGPT 语音、使用 Realtime API 的开发者、在交互式工作流中运行的 agent，以及那些需要在用户仍在说话时处理音频的模型都很重要。

在 OpenAI 的规模下，这转化为三个具体要求：

- 为超过 9 亿周活跃用户提供全球覆盖
- 快速建立连接，让用户在会话开始后几乎立刻就能开口
- 维持低且稳定的媒体往返时延，以及低抖动和低丢包率，让轮流发言足够干脆

负责实时 AI 交互的 OpenAI 团队，最近重新设计了我们的 WebRTC 技术栈，以解决在大规模下开始相互冲突的三个约束：每个会话一个端口的媒体终止方式不适合 OpenAI 的基础设施；有状态的 ICE（交互式连接建立）和 DTLS（数据报传输层安全）会话需要稳定归属；全局路由必须让第一跳延迟保持足够低。在这篇文章中，我们将介绍我们构建的“分离式 relay 加 transceiver”架构，它在保持客户端标准 WebRTC 行为的同时，改变了数据包在 OpenAI 基础设施内部的路由方式。

WebRTC 是一个开放标准，用于在浏览器、移动应用和服务器之间发送低延迟音频、视频和数据。它常被认为与点对点通话相关，但它同样是客户端到服务器实时系统的实用基础，因为它标准化了交互式媒体中最困难的部分：用于建立连通性与 NAT（网络地址转换）穿透的 ICE，用于加密传输的 DTLS 和 SRTP（安全实时传输协议），用于音频压缩与解码的编解码协商，用于质量控制的 RTCP（实时传输控制协议），以及客户端侧的回声消除和抖动缓冲等功能。

这种标准化对 AI 产品很重要。没有 WebRTC，每个客户端都必须分别解决如何跨 NAT 建立连通性、如何加密媒体、如何协商编解码器，以及如何适应不断变化的网络环境。有了 WebRTC，我们就可以建立在浏览器和移动平台已广泛实现的协议栈之上，把自己的工作重心放在把实时媒体连接到模型的基础设施上。

我们也建立在整个 WebRTC 生态之上，包括成熟的开源实现，以及维持浏览器、移动应用和服务器互通的标准化工作。Justin Uberti（WebRTC 最初架构师之一）和 Sean DuBois（Pion 的创建者和维护者）的基础工作，使得像我们这样的团队可以建立在经过实战考验的媒体基础设施之上，而不是重新发明底层传输、加密和拥塞控制行为。我们很幸运，Justin 和 Sean 现在都已成为 OpenAI 的同事，正在帮助我们推动 WebRTC 与实时 AI 更紧密地结合。

对于 AI 来说，最重要的特性是音频以连续流的形式到达。一个语音 agent 可以在用户仍在说话时，就开始转写、推理、调用工具或生成语音，而不必等完整音频上传结束。这就是“像对话一样的系统”和“像按键通话一样的系统”之间的差别。

一旦我们选定了 WebRTC，下一个问题就是在哪里终止它，也就是在哪里接受并拥有这个 WebRTC 连接，比如在边缘节点，以及如何把这些会话连接到推理后端。终止点之所以重要，是因为它决定了我们如何处理实时会话状态、媒体传输、路由、时延和故障隔离。

SFU（Selective Forwarding Unit，选择性转发单元）是一种媒体服务器，它从每个参与方接收一条 WebRTC 流，再有选择地转发给其他参与方。在这种模型里，SFU 为每个参与方终止独立的 WebRTC 连接，而 AI 则作为另一个参与者加入会话。这对本质上就是多人参与的产品来说很合适，比如群组通话、课堂或协作会议。它把音频编解码、RTCP 消息、数据通道、录制和逐流策略都放在同一个位置。

即便是在客户端对 AI 的产品里，SFU 往往也是默认起点，因为它让团队可以复用一套经过验证的系统，用于信令、媒体路由、录制、可观测性，以及未来的扩展，例如人工接管或增加更多参与者。

但我们的工作负载不同。大多数会话是 1:1 的，即一个用户对一个模型，或者一个应用对一个实时 agent，而且每一轮都对时延高度敏感。针对这种流量形态，我们选择了 transceiver 模型：一个 WebRTC 边缘服务终止客户端连接，然后把媒体和事件转换为更简单的内部协议，供模型推理、转写、语音生成、工具调用和编排使用。

在这种设计中，transceiver 是唯一拥有 WebRTC 会话状态的服务，包括 ICE 连通性检查、DTLS 握手、SRTP 加密密钥和会话生命周期。这里的“终止”意味着 transceiver 是完成这些握手并加解密媒体的端点。把这些状态保存在一个地方，让会话所有权更容易推理，也让后端服务能像普通服务一样扩缩容，而不是自己充当 WebRTC 对等体。

在选择 transceiver 模型后，我们的第一版实现是一个基于 Pion 的单一 Go 服务，同时处理信令与媒体终止。它支撑了 ChatGPT 语音、Realtime API 的 WebRTC 端点，以及多个研究项目。

从运维角度看，transceiver 服务承担两项工作：

- 信令：SDP 协商、编解码器选择、ICE 凭据和会话建立
- 媒体：终止下游 WebRTC 连接，并维持到后端推理和编排服务的上游连接

我们希望这个服务像基础设施中的其他服务一样运行在 Kubernetes 上，使工作负载能够根据需求扩缩容，并在不同宿主机间迁移。但传统的“每会话一个端口”的 WebRTC 模式与这种环境并不匹配，因为它依赖大量公开的 UDP 端口范围，而这些端口很难在 pod 被添加、移除或重调度时稳定地暴露、保护和保留。

第一个问题就在于“每会话一个端口”本身。在高并发下，这意味着需要暴露和管理非常大的 UDP 端口范围。

- 云负载均衡器和 Kubernetes Service 并不是为每个服务暴露成千上万个公共 UDP 端口而设计的。每增加一个端口范围，都会给负载均衡配置、健康检查、防火墙策略和发布安全性带来更高复杂度。
- 大范围 UDP 端口很难保障安全，因为它扩大了外部可达的攻击面，也让网络策略更难审计。
- 它也不适合自动扩缩容。Kubernetes 中的 pod 会不断增加、移除和重调度。要求每个 pod 预留并通告一个大的稳定端口范围，会让这种弹性变得脆弱。

这就是为什么许多 WebRTC 系统会转向“每服务器一个 UDP 端口”，然后在这个端口背后做应用层分流。

“每服务器单端口”确实解决了端口数量问题，但它带来了第二个问题：如何在一整个服务集群中保持每个会话的归属。

ICE 和 DTLS 都是有状态协议。创建会话的那个进程，必须持续接收该会话的后续数据包，才能验证连通性检查、完成 DTLS 握手、解密 SRTP，并处理后续的会话变化，例如 ICE 重启。如果同一个会话的数据包落到另一个进程上，建立过程就可能失败，媒体也可能中断。

这让我们的目标变得非常明确：向公网暴露一个小而固定的 UDP 面，同时仍然把每个数据包都路由到拥有对应 WebRTC 会话的 transceiver。

我们评估了几种实现方式，包括 TURN（通过中继绕过 NAT 的穿越方式），即由边缘 relay 终止客户端分配并代表客户端转发流量。

我们最终上线的架构，把数据包路由和协议终止分离开来。信令仍然直接到达 transceiver 以建立会话，而媒体则先进入 relay。relay 是一个轻量级 UDP 转发层，暴露很小的公网入口；transceiver 则是其后的有状态 WebRTC 端点。

relay 不解密媒体，不运行 ICE 状态机，也不参与编解码器协商。它只读取足够的包级元数据来决定目标，再把数据包转发给拥有该会话的 transceiver。transceiver 仍然看到的是正常的 WebRTC 流，仍然拥有全部协议状态。从客户端视角看，WebRTC 会话本身没有任何变化。

在这个架构中，首包路由是关键步骤。relay 必须在会话尚未存在于媒体路径上时，就能正确转发客户端发来的第一个数据包，而不是暂停下来去查询外部查找服务。

每个 WebRTC 会话其实已经自带一个原生的路由钩子：ICE username fragment，也就是 ufrag。它是在会话建立时交换的短标识，并会在 STUN 连通性检查中被回显。我们生成服务端 ufrag 时，会在其中编码足够的路由元数据，让 relay 能推断出目标集群和拥有该会话的 transceiver。

在信令阶段，transceiver 会分配会话状态，并在 SDP answer 中返回共享的 relay VIP 和 UDP 端口。VIP 是位于 relay 集群前面的虚拟 IP；结合端口，它给客户端提供一个稳定目标，例如 `203.0.113.10:3478`，即便这个地址背后实际是许多 relay 实例。客户端在媒体路径上的第一个数据包通常是 STUN binding request，ICE 用它来验证数据包能否到达被通告的地址。

relay 只解析这个首个 STUN 数据包中的必要部分，读取服务端 ufrag，解码路由提示，并把它转发到拥有该会话的 transceiver。每个 transceiver 都监听一个共享 UDP socket，也就是绑定在内部 IP:Port 上的单个操作系统端点，而不是每个会话一个 socket。一旦 relay 建立起从客户端源 IP:Port 到目标 transceiver 的会话映射，后续的 DTLS、RTP 和 RTCP 数据包就会在这个映射中流转，无需再次解码 ufrag。

relay 的会话状态被刻意设计得非常精简，只包含一个内存中的会话映射，用于数据包转发，以及监控所需的计数器和用于会话过期与清理的计时器。这个设计选择让路由逻辑保持在数据包路径之上。如果 relay 重启并丢失这些会话状态，下一个 STUN 数据包会通过 ufrag 中的路由提示重新建立映射。为了进一步提高可靠性，我们还使用 Redis 缓存保存 `<客户端 IP + Port, transceiver IP + Port>` 的映射，一旦路由建立，就能更早恢复，甚至不必等下一个 STUN 包到来。

一旦我们把公网 UDP 暴露面缩减成少量稳定地址和端口，就可以在全球范围内部署同样的 relay 模式。Global Relay 是我们分布式地理部署的 relay 入口集群，它们都实现相同的数据包转发行为。

广泛的地理分布入口可以缩短客户端到 OpenAI 的第一跳，因为数据包可以在离用户更近的地方进入我们的网络，无论是地理位置还是网络拓扑上，而不是先穿过公网到一个遥远区域。实际效果就是更低时延、更少抖动，以及在流量进入我们骨干网络之前更少的可避免丢包突发。

我们使用 Cloudflare 的地理和邻近路由策略来处理信令，使最初的 HTTP 或 WebSocket 请求到达附近的 transceiver 集群。请求上下文决定了会话所在位置，以及向客户端通告哪个 Global Relay 入口点。SDP answer 提供 Global Relay 地址，而 ufrag 则包含足够的信息，让 Global Relay 能把媒体路由到指定集群，再由 relay 路由到目标 transceiver。

地理引导的信令加上 Global Relay，使得信令和媒体都能走一条更近的入口路径，同时让会话仍然锚定在单个 transceiver 上。这降低了信令往返时间，也降低了第一次 ICE 连通性检查的往返时间，直接缩短了用户从开始到可以开口说话之间的等待时间。

我们用 Go 编写了 relay 服务，并有意保持实现范围很窄。在 Linux 上，内核网络栈从机器的网卡接收 UDP 数据包，再把它们投递到 socket，也就是进程在绑定某个 IP:Port 后读取的操作系统端点。relay 运行在用户态，因此一个普通 Go 进程就可以从这个 socket 读取数据包头，更新少量流状态，并在不终止 WebRTC 的情况下完成转发。我们不需要使用任何内核旁路框架，虽然那样可以让用户态进程直接轮询网络队列以获得更高包率，但也会增加运维复杂度。

关键设计选择包括：

- 不做协议终止：relay 只解析 STUN 头和 ufrag；后续的 DTLS、RTP 和 RTCP 都依赖缓存状态转发，使数据包本身保持不透明。
- 短暂状态：它只维护一个小型、短超时的内存映射，把客户端地址映射到 transceiver 目标，用于流状态和可观测性。
- 水平扩展：多个 relay 实例可以并行运行在负载均衡器之后。其状态不是“硬”WebRTC 状态，因此重启只会带来极少量流量损失，并能快速恢复流。

效率优化措施包括：

- `SO_REUSEPORT` 是 Linux 的一个 socket 选项，允许同一台机器上的多个 relay worker 绑定同一个 UDP 端口。随后由内核把到来的数据包分发给这些 worker，避免单个读取循环成为瓶颈。
- `runtime.LockOSThread` 会把每个读取 UDP 的 goroutine 固定在特定 OS 线程上。配合 `SO_REUSEPORT`，这通常能让同一流的数据包，也就是同一组源和目标 IP:Port 加协议，停留在同一个 CPU 核心上，改善缓存局部性并减少上下文切换。
- 预分配缓冲区和最小化拷贝，降低了解析和内存分配开销，从而避免 Go 垃圾回收带来的额外负担。

这个实现以相对较小的 relay 足迹承载了我们的全球实时媒体流量，因此我们保留了这个更简单的设计，而没有走向内核旁路。

这套架构让我们能在 Kubernetes 中运行 WebRTC 媒体，而不必暴露数千个 UDP 端口。这很重要，因为更小且固定的 UDP 暴露面更容易保护和负载均衡，也让基础设施在扩缩容时不必预留大块公共端口范围。借助 Kubernetes 更好的基础设施支持，以及更小攻击面带来的更高安全性，这一设计同时保持了客户端侧的标准 WebRTC 行为，也再次证明了对于我们的工作负载而言，不使用 SFU 才是更合适的默认选择。我们大多数会话都是点对点、对延迟敏感，而且当推理服务不需要表现得像 WebRTC 对等体时，更容易扩展。

更广泛的经验教训是，最适合增加复杂度的地方，是一层很薄的路由层，而不是每个后端服务，也不是客户端的自定义行为。把路由元数据编码进协议原生字段，使我们得以获得可确定的首包路由、小型公网 UDP 暴露面，以及把入口灵活部署到全球更靠近用户的位置。

其中几个选择尤其关键：

- 在边缘保持协议语义不变。客户端依旧说标准 WebRTC，这保证了浏览器和移动端的互操作性。
- 把硬会话状态集中在一个地方。transceiver 拥有 ICE、DTLS、SRTP 和会话生命周期；relay 只负责转发数据包。
- 利用建立阶段中已经存在的信息做路由。ICE ufrag 给了我们一个首包路由钩子，而不需要在热路径上依赖外部查找。
- 先为常见场景做优化，再考虑内核旁路。一个范围受控的 Go 实现，再配合 `SO_REUSEPORT`、线程固定和低分配解析，就足以覆盖我们的工作负载。

实时语音 AI 只有在基础设施让延迟几乎“隐形”时才能成立。对我们来说，这意味着改变 WebRTC 部署的形态，而不改变客户端对 WebRTC 本身的预期。
