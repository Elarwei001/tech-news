### 1. 原生图像生成：fal Provider 和能力显式化 [#49551, #49454]

**PR**: [Image generation: native provider migration and explicit capabilities](https://github.com/openclaw/openclaw/pull/49551) & [Image generation: add fal provider](https://github.com/openclaw/openclaw/pull/49454)  
**作者**: @vincentkoc | **规模**: XL (+2055/-382)

**背景**：OpenClaw 此前依赖 `nano-banana-pro` skill 进行图像生成，这种方式存在局限：skill 层面的抽象无法暴露不同 provider 的能力差异，用户难以了解哪些 provider 支持 image-to-image、多图输出等特性。

**新功能**：

**A. fal 作为内置 Provider**：
```typescript

  id: "fal",
  displayName: "fal.ai",
  
  // 隐式发现
  async shouldImplicitlyEnable(ctx) {
    return ctx.hasEnvVar("FAL_KEY");
  },
  
  // 支持 text-to-image + image-to-image
  capabilities: {
    textToImage: true,
    imageToImage: true,  // FLUX image-to-image
    multipleOutputs: true,
  },
  
  defaultModel: "fal/fal-ai/flux/dev",
};
```

**B. Provider 能力显式化**：
```typescript

  textToImage: boolean;

  aspectRatioControl: boolean;

// 工具调用时校验能力

  request: ImageGenerateRequest,
  provider: ImageGenerationProvider
): ValidationResult {
  if (request.sourceImage && !provider.capabilities.imageToImage) {
    return { valid: false, reason: `${provider.id} does not support image editing` };
  }
  // ...

```

**C. Doctor 自动迁移**：
```typescript

  migrations.push({
    from: "skills.entries.nano-banana-pro",
    to: "agents.defaults.imageGenerationModel",
    newValue: "google/gemini-3-pro-image",  // 或 fal/flux-dev
  });

```

**配置示例**：
```bash

# 或显式配置
```

### 3. Secrets CLI 安全增强：Exec Dry-Run 和审计隔离 [#49417, #49322]

**PR**: [Secrets: gate exec dry-run and preflight resolution behind --allow-exec](https://github.com/openclaw/openclaw/pull/49417) & [CLI: skip exec SecretRef dry-run resolution unless explicitly allowed](https://github.com/openclaw/openclaw/pull/49322)  
**作者**: @joshavant | **规模**: L (+1242/-63)

**背景**：`openclaw config set --dry-run` 和 `openclaw secrets audit` 会解析所有 SecretRef，包括 `exec` 类型。这意味着即使是"只读/预检"操作，也可能触发任意命令执行——这在处理不受信任的配置时构成安全风险。

**新功能**：

**A. Exec 默认跳过**：
```typescript
// src/secrets/exec-resolution-policy.ts
export function selectExecRefs(
  refs: SecretRef[],
  allowExec: boolean
): { toResolve: SecretRef[]; skipped: SecretRef[] } {
  if (allowExec) {
    return { toResolve: refs, skipped: [] };
  }
  
  const execRefs = refs.filter(r => r.source === "exec");
  const nonExecRefs = refs.filter(r => r.source !== "exec");
  
  return {
    toResolve: nonExecRefs,
    skipped: execRefs,
  };
}
```

**B. 三命令统一行为**：
```bash
# config set --dry-run：默认跳过 exec
openclaw config set --batch-file config.json --dry-run
# → skippedExecRefs: ["secrets.providers.vault.token"]

# secrets audit：默认跳过 exec
openclaw secrets audit
# → resolution.skippedExecRefs: ["..."]

# secrets apply --dry-run：默认跳过 exec
openclaw secrets apply --dry-run

# 显式允许 exec（仅 dry-run 场景）
openclaw config set --batch-file config.json --dry-run --allow-exec
```

**C. 非 Dry-Run 保护**：
```typescript
// --allow-exec 只能与 --dry-run 一起使用
if (opts.allowExec && !opts.dryRun) {
  throw new CLIError(
    "--allow-exec is only valid with --dry-run",
    { code: "INVALID_MODE" }
  );
}
```

**D. 审计输出增强**：
```json
{
  "resolution": {
    "refsChecked": 15,
    "skippedExecRefs": ["secrets.providers.vault.token"],
    "resolvabilityComplete": false
  }
}
```

**影响**：安全审计和配置预检不再意外执行命令，符合最小权限原则。

### 2. MS Teams Provider 重复启动导致 EADDRINUSE 崩溃 [#49583]

**PR**: [fix(msteams): prevent EADDRINUSE crash on duplicate provider start](https://github.com/openclaw/openclaw/pull/49583)  
**作者**: @sudie-codes | **规模**: M (+204/-107)

**问题**：Gateway 重启/部署时存在竞态条件，可能触发两次 `monitorMSTeamsProvider()` 调用，导致第二次 `expressApp.listen(port)` 抛出 `EADDRINUSE`，进入无限崩溃-重启循环。

**修复**：Singleton 守卫模式：
```typescript
// extensions/msteams/src/monitor.ts
let activeInstance: Promise<MSTeamsMonitorInstance> | null = null;

export async function monitorMSTeamsProvider(): Promise<MSTeamsMonitorInstance> {
  if (activeInstance) {
    log.warn("MSTeams provider already starting/running, returning existing instance");
    return activeInstance;
  }
  
  activeInstance = doStartProvider().catch(err => {
    activeInstance = null;  // 启动失败时清除，允许重试
    throw err;
  });
  
  return activeInstance;
}

export function resetMSTeamsProviderInstance() {
  activeInstance = null;  // shutdown 或测试隔离时调用
}
```

**影响**：Teams provider 启动更稳健，竞态条件不再导致崩溃循环。

### 5. @YYYYMMDD 模型版本后缀被误解析为 Auth Profile [#48896]

**PR**: [fix(models): preserve @YYYYMMDD version suffixes](https://github.com/openclaw/openclaw/pull/48896)  
**作者**: @Alix-007 | **规模**: L (+566/-50)

**问题**：模型 ID 如 `openai/gpt-4-turbo@20241117` 的 `@20241117` 被当作 auth profile 分隔符，导致模型解析失败。

**修复**：
```typescript
// src/agents/model-ref-profile.ts
const YYYYMMDD_PATTERN = /^@\d{8}$/;

export function parseModelRefWithProfile(input: string): {
  modelRef: string;
  profile?: string;
} {
  const lastAt = input.lastIndexOf("@");
  if (lastAt === -1) {
    return { modelRef: input };
  }
  
  const suffix = input.slice(lastAt);
  
  // 日期版本后缀不是 auth profile
  if (YYYYMMDD_PATTERN.test(suffix)) {
    return { modelRef: input };
  }
  
  // 允许版本后缀 + profile：model@20241117@profile
  return {
    modelRef: input.slice(0, lastAt),
    profile: suffix.slice(1),
  };
}
```

**影响**：`openai/gpt-4-turbo@20241117` 和 `anthropic/claude-3-sonnet@20240229@work` 都能正确解析。
