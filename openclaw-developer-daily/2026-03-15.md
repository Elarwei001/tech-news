### 1. 飞书 ACP/Subagent Session 绑定 [#46819]

**PR**: [feat(feishu): add ACP and subagent session binding](https://github.com/openclaw/openclaw/pull/46819)  
**作者**: @Takhoffman | **规模**: XL (+2988/-48)

**背景**：OpenClaw 在 Discord 和 Telegram 上已支持 ACP（Agent Control Protocol）session 绑定和 subagent 任务委托，但飞书一直落后——无法在飞书 DM 或话题对话中托管持久化 ACP session，子 agent 完成后也无法将结果返回到原始对话。

**新功能**：

1. **飞书 ACP 绑定配置**：`bindings[]` 现在支持 `type: "acp"` + `match.channel: "feishu"`
2. **当前对话 session 绑定**：飞书 DM 和话题对话可通过 `/acp spawn --thread here` 托管绑定 ACP session
3. **Subagent 完成回调**：委托任务完成后自动投递回原始飞书上下文
4. **失败安全**：不支持的通用群聊（非话题）会 fail closed 而非部分绑定

**配置示例**：
```yaml
bindings:
  - type: acp
    match:
      channel: feishu
      conversationId: "oc_xxx"  # 飞书 DM 或话题 ID
    session:
      model: anthropic/claude-opus-4-5
```

**核心实现**（conversation-id 规范化）：
```typescript
// extensions/feishu/src/conversation-id.ts
export function normalizeFeishuConversationId(raw: string): string {
  // 支持多种格式：DM (oc_xxx)、话题 (ot_xxx)、发送者范围话题
  if (raw.startsWith("oc_") || raw.startsWith("ot_")) {
    return raw;
  }
  // 处理复合 ID 格式
  const match = raw.match(/^(oc_|ot_)([a-zA-Z0-9]+)/);
  return match ? `${match[1]}${match[2]}` : raw;
}
```

**Subagent hooks 关键逻辑**：
```typescript
// extensions/feishu/src/subagent-hooks.ts
export async function resolveFeishuDeliveryTarget(
  binding: FeishuBinding,
  context: SubagentCompletionContext
): Promise<DeliveryTarget | null> {
  // 从绑定元数据恢复原始对话目标
  const target = binding.metadata?.deliveryTarget;
  if (!target) return null;
  
  // 验证 Feishu manager 仍然活跃（防止孤儿状态）
```

**影响**：飞书用户现在可以享受与 Discord/Telegram 同等的 ACP 和 subagent 体验——持久化 session、任务委托、结果回调一应俱全。

### 2. OpenRouter 运行时模型能力检测：解决图片静默丢失问题 [#45824]

**PR**: [fix(openrouter): silently dropped images for new OpenRouter models — runtime capability detection](https://github.com/openclaw/openclaw/pull/45824)  
**作者**: @DJjjjhao | **规模**: L (+667/-32)

**背景**：OpenRouter 模型能力由 `@mariozechner/pi-ai/models.generated.ts` 的静态快照决定，构建时固化。任何 pi-ai 发布后新增的 OpenRouter 模型（如 `openrouter/healer-alpha`）不在列表中。

**问题**：未知模型的 fallback 硬编码为 `input: ["text"]`，导致：
- `detectAndLoadPromptImages` 跳过图片加载
- `convertMessages` 丢弃 tool result 中的图片
- **图片被静默丢失**，即使模型实际支持 vision

**静态管道的延迟链**：
```
OpenRouter 新增模型 → pi-mono 手动生成快照 → pi-ai 发布 npm
→ openclaw 更新依赖 → 用户 npm update
```
每一步都需要人工操作，期间所有未知模型图片全丢。

**解决方案**：运行时 API 查询 OpenRouter 模型目录：

```typescript
// src/agents/pi-embedded-runner/openrouter-model-capabilities.ts
const OPENROUTER_CACHE_TTL_MS = 60 * 60 * 1000; // 1 小时缓存

let cachedModels: Map<string, OpenRouterModelCapabilities> | null = null;
let cacheTimestamp = 0;

export async function ensureOpenRouterModelCache(): Promise<void> {
  const now = Date.now();
  if (cachedModels && now - cacheTimestamp < OPENROUTER_CACHE_TTL_MS) {
    return; // 缓存有效
  }
  
  try {
    // ~465KB，通常 1 秒内完成
    const response = await fetch("https://openrouter.ai/api/v1/models");
    const { data } = await response.json();
    
    cachedModels = new Map(
      data.map((m: any) => [m.id, parseCapabilities(m)])
    );
    cacheTimestamp = now;
  } catch (e) {
    // 失败时保持旧缓存或使用 fallback
    console.warn("OpenRouter model cache refresh failed:", e);
  }
}

function parseCapabilities(model: any): OpenRouterModelCapabilities {
  return {
    id: model.id,
    input: model.architecture?.input_modalities ?? ["text"],
    output: model.architecture?.output_modalities ?? ["text"],
    contextWindow: model.context_length ?? 4096,
    supportsReasoning: model.reasoning ?? false,
  };
}
```

**非阻塞预热**：
```typescript
// src/agents/pi-embedded-runner/run.ts
export async function runAgent(ctx: AgentContext): Promise<void> {
  // Fire-and-forget：在 run flow 早期触发
  ensureOpenRouterModelCache().catch(() => {});
  
  // 后续 resolveModel() 时缓存通常已就绪
  // ...
}
```

**性能特点**：
- 首次调用 ~1s，之后 instant（内存缓存 1h）
- API 失败时 graceful degradation 到 `input: ["text"]`（行为同旧版）
- 内置模型和 `models.json` 优先级仍高于 API

**影响**：新 OpenRouter 模型上线即可正常处理图片，不再依赖漫长的静态更新链。

### 1. 低内存主机 Onboarding OOM 崩溃 [#46763]

**PR**: [fix(onboarding): use scoped plugin snapshots to prevent OOM on low-memory hosts](https://github.com/openclaw/openclaw/pull/46763)  
**作者**: @hxy91819 | **规模**: XL (+1127/-99)

**问题**：在 2GB VPS 上运行 `openclaw onboard`，选择 channel 后立即崩溃：
```text
◇  Select channel (QuickStart)
│  Telegram (Bot API)

FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory
```

**根因**：channel 选择后，onboarding 刷新完整 plugin registry，导入并注册**所有**已发现插件——即使只需要选中的那一个。

**修复**：使用作用域化、非激活的 plugin snapshot：
```typescript
// src/plugins/loader.ts
export async function loadPluginSnapshot(
  pluginId: string,
  options: { activate: false }
): Promise<PluginSnapshot> {
  // 只加载指定插件，不修改全局 registry
  const plugin = await discoverPlugin(pluginId);
  return createSnapshot(plugin, { skipGlobalRegistration: true });
}

// src/commands/onboard-channels.ts
async function enableChannelPlugin(channelId: string): Promise<void> {
  // 旧：loadOpenClawPlugins({ cache: false }) — 加载所有插件
  // 新：只加载选中的 channel 插件
  const snapshot = await loadPluginSnapshot(channelId, { activate: false });
  await snapshot.enableForOnboarding();
}
```

**测试验证**：
- 同一台 2GB VPS，修复后顺利完成 Telegram onboarding
- 端到端验证 Telegram 正常工作

**影响**：自托管用户在小内存 VPS 上可以正常完成 onboarding，不再需要调整 `NODE_OPTIONS=--max-old-space-size`。

### 2. Telegram 消息分块断词问题 [#47274]

**PR**: [fix: preserve Telegram word boundaries when rechunking HTML](https://github.com/openclaw/openclaw/pull/47274)  
**作者**: @obviyus | **规模**: M (+242/-6)

**问题**：当 Telegram 消息超出单条长度限制需要分块时，formatter fallback 路径可能在单词中间断开，导致类似：
```
这是一个很长的消息需要被分
割成多条发送
```

**修复**：分块时保留空格/换行断点：
```typescript
function rechunkHtmlPreservingBoundaries(html: string, maxLen: number): string[] {
  const chunks: string[] = [];
  let remaining = html;
  
  while (remaining.length > maxLen) {
    // 在 maxLen 范围内寻找最后一个空白字符
    let splitIdx = remaining.lastIndexOf(' ', maxLen);
    if (splitIdx === -1) splitIdx = remaining.lastIndexOf('\n', maxLen);
    if (splitIdx === -1) splitIdx = maxLen; // fallback
    
    chunks.push(remaining.slice(0, splitIdx).trim());
    remaining = remaining.slice(splitIdx).trim();
  }
  
  if (remaining) chunks.push(remaining);
  return chunks;
}
```

**影响**：Telegram 长消息分块更自然，不会出现断词。
