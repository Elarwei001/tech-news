### 1. Cron 支持自定义 Session 和当前 Session 绑定 [#16511]

**PR**: [feat(cron): support custom session IDs and auto-bind to current session](https://github.com/openclaw/openclaw/pull/16511)  
**作者**: @kkhomej33-netizen | **规模**: L (+617/-118)

**背景**：Cron 任务此前只支持两种模式：`main`（主 session）和 `isolated`（临时隔离 session）。用户无法让定时任务在特定的持久化 session 中执行，也无法方便地绑定到当前对话上下文。

**新功能**：
1. **`"current"` sessionTarget**：从活跃 session（如 Telegram 群组/DM）创建 cron 时，使用 `"current"` 自动绑定到该 session。sessionKey 在创建时解析并存储为 `session:<sessionKey>`。

2. **自定义 session ID**：使用 `"session:project-alpha-daily"` 格式指定持久化命名 session。

**使用示例**：
```json
// 创建绑定到当前频道的每日站会提醒
{
  "name": "Daily Standup",
  "schedule": { "kind": "cron", "expr": "0 9 * * *" },
  "payload": { "kind": "agentTurn", "message": "总结昨天的进展" },
  "sessionTarget": "current"
}

// 使用自定义持久化 session
{
  "name": "Project Monitor",
  "schedule": { "kind": "every", "everyMs": 300000 },
  "payload": { "kind": "agentTurn", "message": "检查项目状态" },
  "sessionTarget": "session:project-alpha-monitor"
}
```

**核心实现**：
```typescript
// cron/types.ts - 扩展 sessionTarget 类型
type CronSessionTarget = "main" | "isolated" | "current" | `session:${string}`;

// cron/service.ts - 创建时解析 "current"
function resolveSessionTarget(
  target: CronSessionTarget,
  currentSessionKey?: string
): CronSessionTarget {
  if (target === "current") {
    if (!currentSessionKey) {
      // CLI/headless 环境无 session 上下文时回退到 isolated
      return "isolated";
    }
    // 将 "current" 解析为实际 sessionKey
    return `session:${currentSessionKey}`;
  }
  return target;
}
```

**UX 改进**：用户不再需要查找并传递 sessionKey——只需使用 `"current"`，系统自动绑定当前上下文。

**向后兼容**：
- 默认行为不变：`systemEvent` → `"main"`，`agentTurn` → `"isolated"`
- 显式设置 `sessionTarget: "current"` 才会启用新功能

**影响**：多 session 架构的 cron 任务现在可以在指定的持久化上下文中运行，保持对话连续性。

### 2. Heartbeat 隔离 Session 模式：Token 成本暴降 95% [#46634]

**PR**: [Heartbeat: add isolatedSession option for fresh session per heartbeat run](https://github.com/openclaw/openclaw/pull/46634)  
**作者**: @odysseus0 | **规模**: S (+148/-13)

**背景**：Heartbeat 运行会在主 session 中累积 turns。随着时间推移（数小时/数天），每次心跳触发时都会重新发送完整对话历史给 LLM，单次消耗约 ~100K tokens。

**问题严重性**：
- 频繁心跳（每 5-60 分钟）= 巨额 token 成本
- 最终触发上下文窗口溢出
- 没有内置逃逸机制

**新功能**：新增 `heartbeat.isolatedSession` 配置项：
```yaml
agents:
  defaults:
    heartbeat:
      every: "5m"
      target: "last"
      isolatedSession: true   # 新增：每次心跳使用全新 session
      lightContext: true      # 建议同时启用
```

**核心实现**：
```typescript
// heartbeat/runner.ts
async function runHeartbeat(ctx: HeartbeatContext): Promise<void> {
  const config = ctx.heartbeatConfig;
  
  // 根据配置决定使用主 session 还是隔离 session
  const sessionKey = config.isolatedSession
    ? `${ctx.sessionKey}:heartbeat`  // 隔离：每次全新
    : ctx.sessionKey;                 // 默认：使用主 session
    
  const session = await resolveCronSession(sessionKey, {
    forceNew: config.isolatedSession  // 强制创建新 session
  });
  
  // 投递路由仍从主 session 读取 lastChannel/lastTo
  const deliveryContext = getDeliveryContext(ctx.mainSession);
  
  await runAgentTurn(session, config.prompt, { deliveryContext });
}
```

**性能对比**：

| 指标 | 非隔离模式 | 隔离模式 | 改善 |
|------|-----------|----------|------|
| 每次心跳 token | ~100K | ~2-5K | **-95%** |
| 上下文窗口风险 | 高 | 无 | ✅ |

**影响**：运行频繁心跳的用户可以将 token 成本降低 95%，同时避免上下文溢出问题。

### 3. /btw 侧边问题：不影响主对话上下文 [#45444]

**PR**: [Add /btw side questions](https://github.com/openclaw/openclaw/pull/45444)  
**作者**: @ngutman | **规模**: XL (+4254/-143)

**背景**：用户希望在主对话进行中快速问一个不相关的问题，但不想污染当前上下文或等待主任务完成。类似 Claude/Pi 的 "by the way" 功能。

**新功能**：
```
/btw 17 * 19 等于多少？只回复数字。
→ 323
```

**核心特性**：
1. **独立于主上下文**：/btw 回答不进入未来的主 agent 上下文
2. **并发安全**：主 session 运行时也能使用 /btw
3. **快照读取**：/btw 读取主 session 最后一个完成分支的快照，不受正在进行的 turn 影响
4. **无工具调用**：/btw 是纯文本回答，不会触发工具执行

**数据流**：
```
主 Session: [Turn 1] → [Turn 2] → [Turn 3 进行中...]
                                      │
/btw 请求 ─────────────────────────────┴──→ 快照 Turn 2 状态
                                               │
                                               ↓
                                        独立 LLM 调用（无工具）
                                               │
                                               ↓
                                        回复（不进入主历史）
```

**核心实现**：
```typescript
// btw.ts
async function runBtwQuestion(
  ctx: SessionContext,
  question: string
): Promise<BtwResult> {
  // 1. 获取 session 快照（不受进行中 turn 影响）
  const snapshot = await ctx.getPreTurnBranchSnapshot();
  
  // 2. 无工具 LLM 调用
  const response = await ctx.provider.complete({
    messages: [...snapshot.messages, { role: "user", content: question }],
    tools: [],  // 关键：禁用工具
  });
  
  // 3. 持久化为非上下文条目
  await ctx.transcriptStore.appendCustomEntry({
    kind: "openclaw:btw",
    question,
    answer: response.content,
    timestamp: Date.now(),
  });
  
  return { answer: response.content };
}
```

**影响**：用户可以在任何时候快速提问，不打断主对话流程，也不污染上下文。

### 飞书线程上下文修复：AI 能看到自己的回复了 [#45254]

**PR**: [fix(feishu): fetch thread context so AI can see bot replies in topic threads](https://github.com/openclaw/openclaw/pull/45254)  
**作者**: @Coobiw | **规模**: L (+483/-37)

**问题**：在飞书话题回复中，AI 只能看到被引用的父消息，看不到自己在该线程中的历史回复——导致多轮对话断裂。

**根因**：`handleFeishuMessage` 只获取 `rootId` 对应的父消息，没有调用飞书 `container_id_type=thread` API 获取完整线程历史。

**修复**：
```typescript
// bot.ts - handleFeishuMessage
if (topicSessionMode && message.rootId) {
  // 1. 获取根消息 → ThreadStarterBody
  const rootMessage = await getMessageFeishu(message.rootId, client);
  
  // 2. 获取线程 ID (omt_xxx 格式)
  const threadId = rootMessage.threadId;
  
  // 3. 新增：获取完整线程历史（包括 bot 回复）
  const threadHistory = await listFeishuThreadMessages(threadId, client);
  
  // 4. 传递给 finalizeInboundContext
  await finalizeInboundContext({
    ThreadStarterBody: rootMessage.text,
    ThreadHistoryBody: threadHistory,  // 新增
    ThreadLabel: threadId,
    MessageThreadId: threadId,
  });
}
```

**新增 API**：
```typescript
// send.ts
async function listFeishuThreadMessages(
  threadId: string,
  client: FeishuClient
): Promise<ThreadMessage[]> {
  const res = await client.im.message.list({
    container_id: threadId,
    container_id_type: "thread",  // 关键参数
  });
  
  return res.items.map(parseThreadMessage);
}
```

**影响**：飞书话题内多轮对话恢复正常，AI 能看到完整上下文。

### 飞书重复回复修复：事件级去重 [#43762]

**PR**: [fix(feishu): add early event-level dedup to prevent duplicate replies](https://github.com/openclaw/openclaw/pull/43762)  
**作者**: @yunweibang | **规模**: M (+210/-45)

**问题**：飞书 webhook 重试和 WebSocket 重连重放可能多次投递同一 `im.message.receive_v1` 事件。现有 `messageId` 去重在 `handleFeishuMessage` 中，在 inbound debouncer 下游执行——两个并发分发可能在任一写入前同时检查缓存，导致 2-3 个重复回复。

**修复**：在 `EventDispatcher` handler 层（`monitor.account.ts`）添加同步内存去重，使用 5 分钟 TTL 和 2000 条目上限：

```typescript
const eventDedup = createDedupeCache({ ttlMs: 5 * 60 * 1000, maxSize: 2_000 });

eventDispatcher.register({
  "im.message.receive_v1": async (data) => {
    const event = data as unknown as FeishuMessageEvent;
    const messageId = event.message?.message_id?.trim();
    if (messageId) {
      const eventKey = `${accountId}:evt:${messageId}`;
      if (!eventDedup.check(eventKey)) {
        log(`feishu[${accountId}]: dropping duplicate event for message ${messageId}`);
        return;
      }
    }
    // ...继续处理
  }
});
```

**影响**：从 Lark SDK 收到事件的最早时刻就去重，彻底消除竞态条件。

## 📊 飞书 Reasoning Stream 支持 [#46029]

**PR**: [feat(feishu): add reasoning stream support to streaming cards](https://github.com/openclaw/openclaw/pull/46029)  
**作者**: @day253 | **规模**: S (+168/-9)

当 reasoning level 设为 `stream`（`/reasoning stream`）时，thinking tokens 现在渲染为 markdown blockquote，与答案之间用 `---` 分隔：

```markdown
> 💭 **Thinking**
> 让我分析用户的请求...
> 他们想用 seedream 生成图片...

---

这是生成的图片！我用了 seedream API...
```

**核心实现**：
```typescript
// reply-dispatcher.ts
function buildCombinedStreamText(): string {
  const parts: string[] = [];
  
  if (reasoningBuffer.length > 0) {
    parts.push(`> 💭 **Thinking**\n> ${reasoningBuffer.join("\n> ")}`);
    parts.push("---");
  }
  
  if (answerBuffer.length > 0) {
    parts.push(answerBuffer);
  }
  
  return parts.join("\n\n");
}
```

**向后兼容**：默认 reasoning level 是 `off`——无行为变化。
