### 1. Plugin-SDK 构建优化：内存占用暴降 63% [#45426]

**PR**: [perf(build): deduplicate plugin-sdk chunks to fix ~2x memory regression](https://github.com/openclaw/openclaw/pull/45426)  
**作者**: @TarasShyn | **规模**: S (+86/-6)

**背景**：OpenClaw 3.12 版本出现严重的内存回退问题。社区用户报告 gateway 启动后内存占用翻倍，甚至导致低配 VPS 和 Raspberry Pi 无法运行。

**根因分析**：tsdown 对 38 个 plugin-sdk 入口点分别构建，阻止了 bundler 共享公共 chunk。结果是大量代码重复：
- `query-expansion` 模块被复制 20 次
- `fetch` 模块被复制 14 次
- `logger` 模块被复制 11 次

**修复方案**：单次构建通道处理所有 38 个入口点：
```javascript
// tsdown.config.js - 修复后
export default {
  entry: [
    "src/plugin-sdk/index.ts",
    "src/plugin-sdk/browser.ts",
    "src/plugin-sdk/memory.ts",
    // ... 所有 38 个入口
  ],
  splitting: true,  // 启用代码分割
  format: "esm",
  // 关键：单次构建 = 自动共享 chunk
};
```

**性能对比**：

| 指标 | 修复前 | 修复后 | 改善 |
|------|--------|--------|------|
| `dist/` 大小 | 190MB | 64MB | **-66%** |
| `plugin-sdk/` 大小 | 142MB | 16MB | **-89%** |
| JS 文件数 | 1,395 | 789 | **-43%** |
| 5MB+ 文件数 | 27 | 7 | **-74%** |
| Plugin-SDK 堆内存 | +1,309MB | +63MB | **-95%** |
| 总堆内存 | 1,926MB | 711MB | **-63%** |

**影响**：低配设备用户终于可以正常运行 OpenClaw，Raspberry Pi 和入门级 VPS 部署不再受阻。

### 2. Slack Interactive Replies：按钮和下拉菜单支持 [#44607]

**PR**: [Slack: add opt-in interactive reply directives](https://github.com/openclaw/openclaw/pull/44607)  
**作者**: @vincentkoc | **规模**: L (+893/-10)

**背景**：社区长期请求 Slack 快速操作支持（#12602）。Telegram 早已有 inline buttons，但 Slack 用户只能用纯文本交互。

**新功能**：Agent 可以通过指令语法生成 Slack Block Kit 组件：

```markdown
# 按钮指令
[[slack_buttons: 确认:confirm, 取消:cancel, 删除:delete]]

# 下拉菜单指令
[[slack_select: 选择操作 | 编辑:edit, 复制:copy, 删除:delete]]
```

**渲染效果**：
```json
{
  "blocks": [
    {
      "type": "actions",
      "elements": [
        { "type": "button", "text": {"type": "plain_text", "text": "确认"}, "action_id": "confirm" },
        { "type": "button", "text": {"type": "plain_text", "text": "取消"}, "action_id": "cancel" },
        { "type": "button", "text": {"type": "plain_text", "text": "删除"}, "action_id": "delete", "style": "danger" }
      ]
    }
  ]
}
```

**配置启用**：
```yaml
channels:
  slack:
    capabilities:
      interactiveReplies: true  # 默认关闭，需显式开启
```

**核心实现**：
```typescript
// reply-normalize.ts
function compileSlackDirectives(text: string, enabled: boolean): SlackBlocks | null {
  if (!enabled) return null;
  
  const buttonMatch = text.match(/\[\[slack_buttons:\s*(.+?)\]\]/);
  if (buttonMatch) {
    const choices = parseChoices(buttonMatch[1]);
    return {
      blocks: [{
        type: "actions",
        elements: choices.map(({ label, value }) => ({
          type: "button",
          text: { type: "plain_text", text: label },
          action_id: value
        }))
      }]
    };
  }
  // ... 类似处理 slack_select
}
```

**Agent Prompt Hints**：系统会根据配置状态自动注入提示：
```
# 当 interactiveReplies: true 时
You can use [[slack_buttons: Label:value, ...]] for quick actions.

# 当 interactiveReplies: false 时
Interactive Slack components are disabled in this workspace.
```

**影响**：Slack 用户现在可以获得类似 Telegram 的交互体验，显著提升工作流效率。

## 🐛 重要 Bugfix

### Gateway 启动崩溃：Anthropic 模型引用 TDZ 问题 [#45520]

**PR**: [fix(config): avoid Anthropic startup crash](https://github.com/openclaw/openclaw/pull/45520)  
**作者**: @BunsDev | **规模**: S (+48/-8)

**问题**：用户配置文件中包含 `anthropic/claude-sonnet-4-20250514` 这样的带日期模型引用时，gateway 启动直接崩溃：

```
ReferenceError: Cannot access 'ANTHROPIC_MODEL_ALIASES' before initialization
```

**根因**：`applyContextPruningDefaults()` 在配置加载时调用 `parseModelRef()`，后者又调用 `normalizeAnthropicModelId()`。但在打包后的启动顺序中，别名表还在 temporal dead zone (TDZ) 中。

**修复**：直接在函数内部解析别名，不依赖模块级常量：
```typescript
// model-selection.ts
export function normalizeAnthropicModelId(modelId: string): string {
  // 修复前：读取模块级 ANTHROPIC_MODEL_ALIASES（TDZ 风险）
  // 修复后：直接内联处理
  const shorthandAliases: Record<string, string> = {
    "opus": "claude-opus-4-5",
    "sonnet": "claude-sonnet-4-6",
    // ...
  };
  
  if (shorthandAliases[modelId]) {
    return shorthandAliases[modelId];
  }
  
  // 带日期的模型 ID 保持不变
  if (/^claude-.*-\d{8}$/.test(modelId)) {
    return modelId;
  }
  
  return modelId;
}
```

**影响**：使用带日期版本 Anthropic 模型的用户不再遭遇启动崩溃。

### 跨 Agent Workspace 解析错误 [#40176]

**PR**: [fix: resolve target agent workspace for cross-agent subagent spawns](https://github.com/openclaw/openclaw/pull/40176)  
**作者**: @moshehbenavraham | **规模**: M (+234/-10)

**问题**：PM Agent 通过 `sessions_spawn(agentId: "ops")` 启动 Ops Agent 时，子 agent 错误地继承了 PM 的 workspace 和 persona 文件，而非 Ops 自己的。

**根因**：`resolveSpawnedWorkspaceInheritance` 始终使用调用者的 workspace，没有考虑 `targetAgentId`：

```typescript
// 修复前
function resolveSpawnedWorkspaceInheritance(ctx: Context): WorkspacePath {
  // 总是返回 ctx.workspaceDir (调用者的 workspace)
  return ctx.workspaceDir ?? resolveDefaultWorkspace(ctx.agentId);
}

// 修复后
function resolveSpawnedWorkspaceInheritance(
  ctx: Context, 
  targetAgentId?: string
): WorkspacePath {
  // 跨 agent spawn：解析目标 agent 的 workspace
  if (targetAgentId && targetAgentId !== ctx.agentId) {
    return resolveDefaultWorkspace(targetAgentId);
  }
  // 同 agent spawn：继承调用者的 workspace
  return ctx.workspaceDir ?? resolveDefaultWorkspace(ctx.agentId);
}
```

**调用点修复**：
```typescript
// subagent-spawn.ts
const targetWorkspace = resolveSpawnedWorkspaceInheritance(
  ctx,
  params.agentId  // 传入目标 agent ID
);

// 跨 agent spawn 时不传递调用者的 explicitWorkspaceDir
if (params.agentId !== ctx.agentId) {
  spawnOptions.explicitWorkspaceDir = undefined;
}
```

**影响**：多 agent 架构（PM/Ops/Dev 分工）现在能正确隔离各自的 workspace 和 persona。

### Cron 隔离任务死锁 [#45459]

**PR**: [fix(cron): prevent isolated cron nested lane deadlocks](https://github.com/openclaw/openclaw/pull/45459)  
**作者**: @vincentkoc | **规模**: XS (+49/-0)

**问题**：`sessionTarget: "isolated"` 的 cron job 在执行嵌套工作（如 compaction）时可能无限等待——因为内部工作试图排队到同一个 cron lane，但该 lane 已被外层任务占用。

**修复**：embedded runner 的 lane 解析中，将 `cron` 映射为 `nested`：
```typescript
// lanes.ts
export function resolveGlobalLane(lane: CommandLane): CommandLane {
  switch (lane) {
    case CommandLane.Cron:
      return CommandLane.Nested;  // 避免死锁
    case CommandLane.Session:
      return CommandLane.Session;
    default:
      return lane;
  }
}
```

**影响**：使用 isolated cron 的用户不再遇到任务无响应超时。

## 🔒 安全加固

### Hook 加载器路径验证 [#44437]

**PR**: [fix(hooks): fail closed on unreadable loader paths](https://github.com/openclaw/openclaw/pull/44437)  
**作者**: @vincentkoc | **规模**: S (+99/-19)

之前 hook 模块路径解析失败时会 fail-open（回退到 `path.resolve`）。现在改为 fail-closed，同时添加可信代码加载警告：

```typescript
// loader.ts
async function resolveHookModulePath(base: string, module: string): Promise<string> {
  try {
    const realPath = await fs.realpath(path.join(base, module));
    return realPath;
  } catch (err) {
    // 修复前：return path.resolve(base, module);  // fail-open
    // 修复后：
    throw new HookLoadError(`Cannot resolve hook module: ${module}`);
  }
}
```

## 📊 Compaction 系统改进

### Post-Compaction Token 计数修复 [#28347]

**PR**: [fix(compaction): use full-session token count for post-compaction sanity check](https://github.com/openclaw/openclaw/pull/28347)  
**作者**: @efe-arv | **规模**: S (+142/-5)

**问题**：Compaction 后状态显示 `Context ?/200k` 而非实际值如 `Context 42k/200k (21%)`。

**根因**：sanity check 比较的是两个不同范围的值：
- `tokensAfter`：完整 session（系统提示 + 上下文 + 总结 + 历史）
- `tokensBefore`：仅可总结的历史子集

对于大系统提示（15-30k tokens），`tokensAfter > tokensBefore` 几乎总是成立，导致结果被丢弃。

### Persona 和语言连续性保持 [#10456]

**PR**: [fix: preserve persona and language continuity in compaction summaries](https://github.com/openclaw/openclaw/pull/10456)  
**作者**: @keepitmello | **规模**: M (+326/-2)

**问题**：韩语 persona agent 在 compaction 后突然切换成英语回复。

**根因**：SDK 的 `autoCompact()` 硬编码 `customInstructions: undefined`，总结提示是纯英语，导致总结输出也是英语。

**修复**：新增配置项 `compaction.customInstructions`，默认值指示总结器保持对话语言。
