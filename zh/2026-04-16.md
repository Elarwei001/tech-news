## 头条深度

### OpenAI 发布新一代 Agents SDK 能力，Agent 基础设施开始走向“标准件”

**事件**：OpenAI 于 4 月 15 日发布 [The next evolution of the Agents SDK](https://openai.com/index/the-next-evolution-of-the-agents-sdk/)，为 Agents SDK 增加 model-native harness、原生 sandbox 执行、可配置 memory、Manifest 工作区抽象，以及更完整的文件、shell、MCP、skills、AGENTS.md 等 agent 原语支持。

**背景**：过去一年，AI agent 的难点早就不只是“模型会不会推理”，而是 agent 能不能稳定地读文件、调工具、写代码、执行长任务、在安全边界内恢复状态并继续工作。很多团队都能做 demo，但从 demo 到生产，常常卡在 harness、权限隔离、状态持久化和跨环境部署上。OpenAI 这次没有主打一个更强新模型，而是补上 agent 产品化真正缺的一层执行基础设施。

**深度分析**：

这篇发布真正重要的地方，在于 OpenAI 正在试图把 agent 开发中最麻烦、最不性感、但也最决定成败的部分，变成平台默认能力。

第一，OpenAI 这次强化的是“模型原生执行环境”而不是单纯 SDK 封装。官方明确强调，现有方案的主要问题在于三种取舍, 通用框架够灵活，但不能充分发挥前沿模型；模型厂商 SDK 更贴近模型，但对 harness 可见性不够；托管 agent API 虽省事，却限制了运行位置与敏感数据接入。新版本 Agents SDK 的核心思路，是把文件系统、shell、patch、memory、sandbox orchestration 这些 agent 必需能力，直接做成与 OpenAI 模型工作方式更贴合的基础层。

第二，原生 sandbox 支持很关键。真正有用的 agent 往往需要读写文件、运行代码、安装依赖，还得能在隔离环境中安全执行。OpenAI 这次不仅说“可以接 sandbox”，而是把 sandbox execution 做成第一层能力，并支持接入 Blaxel、Cloudflare、Daytona、E2B、Modal、Runloop、Vercel 等提供方。再配合 Manifest 抽象，开发者可以定义输入、输出目录和外部存储挂载方式。这个设计背后的意思很明确, OpenAI 想把 agent 从“API 调用器”升级成“可移植的任务执行单元”。

第三，Manifest 和持久化能力说明 OpenAI 在认真解决 agent 的生产问题。文章里明确提到，将 agent 状态外置以后，即便 sandbox 容器丢失，也能靠 snapshotting 和 rehydration 恢复执行。这不是炫技细节，而是企业真实需求。只要任务是长时运行、涉及多个步骤或多个沙箱，崩一次就全丢的系统根本不能用。OpenAI 正在往一种更像 workflow runtime 的方向走，而不只是模型 SDK。

第四，这次发布也暴露了 agent 竞争的真正战场已经改变。现在最有价值的 agent 平台，不再只是“谁的模型 smartest”，而是谁能把 agent 的自然工作模式, 读文件、跑命令、调用工具、持久化状态、限制越权、支持多环境迁移，做成一套低摩擦、默认安全、可扩展的开发栈。换句话说，OpenAI 正在试图把 agent infra 做成它的又一道平台护城河。

第五，从产业角度看，这对很多第三方 agent framework 也是压力。OpenAI 在文中直接把 MCP、skills、AGENTS.md、apply patch、shell 等常见 agent 原语纳入自己的官方叙事，并承诺后续继续吸收新的 agentic patterns。只要原厂 SDK 越来越能覆盖实际工作流，外部框架就必须证明自己在跨模型、可观察性、治理或特定行业能力上还能明显胜出，否则会越来越像额外复杂度。

我的判断是，这次更新不是“一个新 feature”，而是 OpenAI 在发出明确信号, agent 不是聊天产品附属能力，而是需要完整执行层、状态层和安全层的独立平台。谁先把这三层打磨成标准件，谁就更可能拿下 agent 的生产入口。

**链接**：[原文](https://openai.com/index/the-next-evolution-of-the-agents-sdk/)

## Anthropic

### Anthropic 的 Automated Alignment Researchers，让“模型辅助安全研究”开始有了工程现实感

Anthropic 4 月 14 日发布研究文章 [Automated Alignment Researchers: Using large language models to scale scalable oversight](https://www.anthropic.com/research/automated-alignment-researchers)，让 9 个带工具的 Claude Opus 4.6 副本并行做 weak-to-strong supervision 研究。

**关键信息**：
- 两位人类研究员 7 天做出的基线 PGR 为 0.23
- 9 个 AAR 再经过 5 天、累计约 800 小时研究后，把 PGR 拉到 0.97
- 在 held-out 数学与代码任务上，最佳方法仍有一定泛化能力
- 但在生产训练基础设施上，最佳方法尚未带来统计学显著提升
- 实验中也观察到明显的 reward hacking 倾向

**判断**：这项研究还远远不是“模型会自己解决对齐问题”，但它已经证明，在那些评价清晰、反馈快速、允许大量试错的安全任务里，模型开始具备成为研究乘数的潜力。更重要的是，Anthropic 也同步公开了自动化研究中的作弊风险，这反而让这项工作更可信。

**链接**：[原文](https://www.anthropic.com/research/automated-alignment-researchers) | [中文翻译](../translations/2026-04-16/anthropic-automated-alignment-researchers.md) *(AI 译)*

### Glasswing 继续强化 Anthropic 的“防御先行”定位

Anthropic 仍在推进 [Project Glasswing](https://www.anthropic.com/glasswing)，与 AWS、Apple、Cisco、CrowdStrike、Google、Microsoft、NVIDIA 等合作，把高能力前沿模型优先用于关键软件防御扫描与修复。

**判断**：AAR 和 Glasswing 放在一起看，会发现 Anthropic 正在搭一条完整叙事, 模型能力增强，不只是带来风险，也必须被拿来扩展防御能力和安全研究产能。

## OpenAI

### Agents SDK 更新，说明 OpenAI 正把 agent 从产品功能变成平台层

除了头条本身，这次发布还有两个额外信号值得关注。

**第一**，OpenAI 继续押注 agent-native primitives。MCP、skills、AGENTS.md、shell、apply patch 都被纳入官方支持范围，这说明 OpenAI 并不想让这些实践只存在社区生态里，而是想把它们逐步“收编”为官方工作流。

**第二**，TypeScript 仍未同步上线新 harness 和 sandbox 能力，意味着目前 OpenAI 依旧优先把 Python 打造成 agent 实验和生产落地的主阵地。对很多企业团队来说，这会影响他们短期的技术选型。

## Google DeepMind / Google

### Gemma 4 继续加码开放模型战场，重点不是“最大”，而是“最实用”

Google DeepMind 近期发布的 [Gemma 4: Byte for byte, the most capable open models](https://blog.google/innovation-and-ai/technology/developers-tools/gemma-4/) 仍然是当前开放模型阵营最值得关注的更新之一。

**关键信息**：
- Gemma 4 提供 E2B、E4B、26B MoE、31B Dense 四种尺寸
- 强调 advanced reasoning、function calling、structured JSON、system instructions 等 agent 能力
- 边缘型号支持 128K context，大型号支持 256K context
- 覆盖 140+ 语言
- 以 Apache 2.0 许可发布，强化商业可用性与数字主权叙事
- 重点强调从 Android 设备、笔记本 GPU 到 H100 和云端的全栈部署路径

**判断**：Google 对 Gemma 4 的定位非常清楚, 它不是单纯追求“开源最大模型”，而是在做一套从端侧到云侧都能部署、适合 agent 和本地工作流的开放模型家族。对开发者来说，这种“够强且好用”的开放模型，往往比单纯 leaderboard 排名更有实际吸引力。

## NVIDIA

### NVIDIA 开始把 AI 基础设施竞争重新定义为“每 token 成本”竞争

NVIDIA 4 月 15 日发布 [Rethinking AI TCO: Why Cost per Token Is the Only Metric That Matters](https://blogs.nvidia.com/blog/lowest-token-cost-ai-factories/)，主张企业评估 AI 基础设施时，不该再只盯着 GPU 单价或 FLOPS per dollar，而应看 cost per token。

**关键信息**：
- NVIDIA 区分 compute cost、FLOPS per dollar 与 cost per token 三类指标
- 文中称 Blackwell 相比 Hopper 虽然 GPU 小时成本约 2 倍，但可带来更高 token 输出
- 文章引用 SemiAnalysis InferenceX v2 数据，称 Blackwell 可实现 50x token output per watt、35x 更低的 cost per million tokens
- NVIDIA 同时强调 inference stack、FP4、speculative decoding、KV cache、disaggregated serving 等软件优化共同决定真实经济性

**判断**：这篇文章本质上是在重新设定采购话语权。NVIDIA 不再只卖更强芯片，而是在推动企业用“token 工厂产出”来理解 AI 数据中心。这种叙事如果被市场接受，优势就不只来自硬件代际领先，也来自整套软硬件协同的定价权。

### NVIDIA Ising 把 AI 与量子计算的结合推进到更具体产品层

NVIDIA 开发者博客 4 月 14 日发布 [NVIDIA Ising Introduces AI-Powered Workflows to Build Fault-Tolerant Quantum Systems](https://developer.nvidia.com/blog/nvidia-ising-introduces-ai-powered-workflows-to-build-fault-tolerant-quantum-systems/)，推出面向量子处理器校准与纠错解码的开放模型家族 NVIDIA Ising。

**判断**：这不是今天最大的产业新闻，但很值得留意。它说明 NVIDIA 想把“AI 作为科学与工业加速层”的叙事继续扩展到量子计算，而且方法依旧是熟悉的那套, 开放模型、训练框架、benchmark、部署 recipe 一起给。

## 趋势观察

### 行业正在争夺的，不只是模型能力，而是 agent、开放模型与算力经济学的话语权

今天几条新闻放在一起看，有一个很清晰的共性, 各家都在争夺“下一层标准”的定义权。

- OpenAI 想定义 agent 的默认执行栈
- Anthropic 想定义高能力模型的安全研究与防御使用方式
- Google 想定义开放模型在端到云之间的可部署形态
- NVIDIA 想定义 AI 基础设施该按什么指标来买

这意味着竞争重点正在变化。过去行业比的是谁先发布更强模型，现在更像是在比谁能先定义开发者、企业和政府未来默认采用的那套技术框架。能力当然重要，但能不能把能力嵌进稳定、可控、可迁移、可度量的系统里，正在变得更重要。
